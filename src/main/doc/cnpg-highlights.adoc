== Was ist CloudNativePG?

[.lead]
Der *einzige* PostgreSQL-Operator, der vom CNCF offiziell angenommen wurde +
und PostgreSQL als *First-Class-Citizen* in Kubernetes behandelt.

[cols="1,1,1",frame=none,grid=none]
|===
| icon:check-circle[role=green] *CNCF Sandbox Projekt* +
seit Mai 2022
| icon:check-circle[role=green] *Zalando-Erbe* +
weiterentwickelt als echter K8s-Operator
| icon:check-circle[role=green] *Produktionsreif* +
ab Version 1.15
|===

[NOTE.speaker]
--
CNPG wurde von EDB (EnterpriseDB) entwickelt und 2022 der CNCF gespendet.
Im Gegensatz zu Spilo/Patroni lebt die Operator-Logik *außerhalb* des Datenbankcontainers.
--

== Architektur: Operator vs. Spilo

[cols="~,~",options="header"]
|===
| Spilo / Zalando Operator | CloudNativePG

| Patroni *im* Container +
(Sidecar-Architektur)
| Operator *außerhalb* — reiner K8s Controller

| Konfiguration via +
Env-Variablen + `SPILO_CONFIGURATION`
| Deklarative `Cluster`-CRD +
(vollständig GitOps-fähig)

| PodEnvSource + ConfigMap-Mischung
| Single Source of Truth: +
`kubectl get cluster`

| Failover via Patroni DCS +
(etcd / Kubernetes)
| Failover via K8s Watch Loop +
kein externer DCS notwendig

| Backup: WAL-G, extern konfiguriert
| Backup: Barman Object Store, +
nativ in der CRD
|===

== Kernfeature: Deklarative Cluster-CRD

[source,yaml]
----
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: production-db
spec:
  instances: 3                                    # <1>
  imageName: ghcr.io/cloudnative-pg/postgresql:17

  postgresql:
    parameters:
      max_connections: "500"
      shared_buffers: "2GB"

  storage:
    size: 200Gi
    storageClass: fast-nvme

  backup:                                         # <2>
    barmanObjectStore:
      destinationPath: s3://backups/production-db
      retentionPolicy: "30d"

  managed:                                        # <3>
    roles:
      - name: appuser
        login: true
        passwordSecret:
          name: appuser-credentials
----

<1> Primary + 2 Standbys, automatisch verwaltet
<2> WAL-Archivierung + Base Backups in S3/GCS/Azure
<3> Datenbankrollen deklarativ — kein manuelles `CREATE ROLE`

== Backup & Recovery

=== Barman Object Store — nativ integriert

[cols="1,2"]
|===
| Feature | Detail

| *WAL-Archivierung*
| Kontinuierlich, komprimiert (gzip/bzip2/snappy), verschlüsselt

| *Base Backups*
| Geplant via `ScheduledBackup` CRD +
`0 2 * * *` → täglich 02:00 Uhr

| *Point-in-Time Recovery*
| `targetTime`, `targetLSN`, `targetXID` +
Bootstrap eines neuen Clusters aus beliebigem Zeitpunkt

| *Retention*
| Deklarativ: `"7d"`, `"4w"`, `"12m"`

| *Object Stores*
| AWS S3, GCS, Azure Blob, MinIO, Garage
|===

[source,yaml]
----
# PITR: Wiederherstellung auf exakten Zeitpunkt
spec:
  bootstrap:
    recovery:
      source: prod-backup
      recoveryTarget:
        targetTime: "2026-03-15 14:30:00"
----

== Migration von Spilo: spilo2cnpg

=== Der Migrations-Workflow

[source,bash]
----
# 1. Analyse: Blocker und Warnungen identifizieren
java -jar spilo2cnpg.jar analyze \
  --all-namespaces \
  --output json | jq '.[] | select(.readiness == "BLOCKED")'

# 2. Manifest generieren (WAL-Recovery für minimale Downtime)
java -jar spilo2cnpg.jar generate myapp-db.yaml \
  --strategy WAL_RECOVERY \
  --s3-bucket my-pg-backups \
  --output cnpg-cluster.yaml

# 3. CNPG-Cluster bootstrappen (recovered aus WAL-Archiv)
kubectl apply -f cnpg-cluster.yaml

# 4. Warten bis Cluster Ready
kubectl wait cluster/myapp-db \
  --for=condition=Ready --timeout=10m
----

== Connection Pooling: Pooler CRD

[source,yaml]
----
apiVersion: postgresql.cnpg.io/v1
kind: Pooler
metadata:
  name: production-db-pooler-rw
spec:
  cluster:
    name: production-db
  instances: 3
  type: rw                          # rw | ro
  pgbouncer:
    poolMode: transaction           # session | transaction | statement
    parameters:
      max_client_conn: "1000"
      default_pool_size: "25"
----

* PgBouncer als *eigenständige CRD* — nicht im DB-Container
* Unabhängig skalierbar (`instances: 3`)
* *Separater Service* — Anwendungen zeigen auf Pooler, nicht direkt auf Cluster

== pg_tde: Transparente Datenverschlüsselung

=== Encryption at Rest — ohne Filesystem-Tricks

[source,yaml]
----
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: encrypted-db
spec:
  imageName: ghcr.io/cloudnative-pg/postgresql:17-tde  # <1>
  postgresql:
    parameters:
      tde_encryption_key_provider: "file"               # <2>
      tde_encryption_key_file: "/etc/tde/key"
----

<1> Spezielles Image mit `pg_tde` Extension vorinstalliert
<2> Key Provider: file, Vault (HashiCorp), KMIP

*Vorteile gegenüber Filesystem-Verschlüsselung:*

* Verschlüsselung auf WAL- und Heap-Ebene
* Key Rotation ohne Datenbankausfall
* Keine Abhängigkeit vom Storage-Provider

== kubectl cnpg Plugin

=== Administration direkt aus der CLI

[source,bash]
----
# Status aller Cluster im Namespace
kubectl cnpg status production-db

# Sofortiger Switchover (Zero-Downtime Maintenance)
kubectl cnpg promote production-db standby-1

# Manuelles Backup auslösen
kubectl cnpg backup production-db

# Verbindungsstring für Anwendung ausgeben
kubectl cnpg psql production-db -- -c "SELECT version();"

# Cluster-Konfiguration validieren (Dry-Run)
kubectl cnpg validate production-db

# Log-Streaming aller Cluster-Pods
kubectl cnpg logs production-db
----

* Installation: `kubectl krew install cnpg`
