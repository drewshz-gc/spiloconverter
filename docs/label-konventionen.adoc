= Label-Konventionen CloudNativePG (CNPG)
:toc:
:toclevels: 3

Stand: 2026-06-17 +
Kontext: Migration Spilo/Zalando -> CNPG, abgeleitet aus Pilot `tifa01-cnpg-cluster` +
Bezug: CCPSQL-130 (Namenskonventionen), Diskrepanz NetworkPolicies
(siehe `diskrepanzen-zalando-cnpg.adoc`)

== Ziel

Ein einheitliches, verbindliches Label-Schema fuer alle CNPG-bezogenen
Ressourcen, damit:

* NetworkPolicies, ServiceMonitors, Kustomize/ArgoCD und Backups *verlaesslich*
  selektieren koennen,
* Tenant/Partition/Umgebung maschinell auswertbar sind,
* die Herkunft (Migration aus Spilo) nachvollziehbar bleibt.

== Zwei Kategorien von Labels

=== A) Operator-verwaltete Labels (`cnpg.io/*`) -- NICHT selbst setzen

Werden vom CNPG-Operator automatisch auf Pods/Jobs/PVCs gesetzt. *Nur lesen*
(z.B. fuer NetworkPolicy-Selektoren), nicht manuell pflegen.

[cols="2,3"]
|===
| Label | Bedeutung / Werte

| `cnpg.io/cluster`       | Cluster-Name (auf allen Pods/PVCs/Jobs des Clusters)
| `cnpg.io/instanceName`  | konkrete Instanz (z.B. `tifa01-cnpg-cluster-1`)
| `cnpg.io/podRole`       | `instance`
| `cnpg.io/instanceRole`  | `primary` / `replica`
| `cnpg.io/jobRole`       | `import` / `initdb` / `join` / `pgbasebackup` / `full-recovery`
| `cnpg.io/poolerName`    | Name des zugehoerigen Pooler (auf Pooler-Pods)
|===

NOTE: Fuer Pod-Selektoren (NetworkPolicy) ist `cnpg.io/cluster` der
zuverlaessigste Anker, weil der Operator ihn garantiert auf *allen* Pods setzt.

=== B) Eigene Labels (von uns gesetzt / generiert)

Auf den *Manifesten* (Cluster, Database, ScheduledBackup, CronJob, Pooler,
ArgoCD-Application) sowie -- via `inheritedMetadata` -- auf den daraus
erzeugten Pods/Services/PVCs.

[cols="2,2,3"]
|===
| Label | Beispielwert | Quelle / Regel

| `app.kubernetes.io/name`       | `postgresql`            | fix
| `app.kubernetes.io/instance`   | `tifa01-cnpg-cluster`   | = Cluster-Name
| `app.kubernetes.io/part-of`    | `tifa`                  | = Tenant
| `app.kubernetes.io/component`  | `database` \| `pooler` \| `backup` \| `logical-backup` | je Ressourcentyp
| `app.kubernetes.io/managed-by` | `spilo2cnpg` \| `argocd`| Erzeuger
| `tenant`                       | `tifa`                  | Tenant
| `partition`                    | `p03-stg`               | Partition/Cluster-Standort
| `environment`                  | `prod` \| `stg`         | Umgebung
| `spilo2cnpg/source-cluster`    | `tifa01-postgres-cluster` | urspruenglicher Spilo-Name (Migrations-Trace)
|===

== Schluesselmechanismus: `inheritedMetadata`

Damit unsere Labels (Kategorie B) auch auf den vom Operator erzeugten
*Pods/Services/PVCs* landen -- und somit fuer NetworkPolicies selektierbar
sind -- werden sie im Cluster unter `spec.inheritedMetadata` gesetzt:

[source,yaml]
----
spec:
  inheritedMetadata:
    labels:
      app.kubernetes.io/name: postgresql
      app.kubernetes.io/instance: tifa01-cnpg-cluster
      app.kubernetes.io/part-of: tifa
      app.kubernetes.io/managed-by: spilo2cnpg
      tenant: tifa
      partition: p03-stg
      environment: stg
      spilo2cnpg/source-cluster: tifa01-postgres-cluster
----

WICHTIG: Ohne `inheritedMetadata` tragen die Pods nur die `cnpg.io/*`-Labels --
eigene Tenant-/Partition-Selektoren in NetworkPolicies wuerden dann nicht
greifen.

== Label-Matrix je Ressourcentyp

[cols="2,3"]
|===
| Ressource | `app.kubernetes.io/component`

| `Cluster`               | (kein component; Basis-Labels) + `inheritedMetadata`
| `Database`              | `database`
| `Pooler`                | `pooler`
| `ScheduledBackup`       | `backup`
| `CronJob` (logical)     | `logical-backup`
| `Backup`                | `backup`
| ArgoCD `Application`    | `database` (plus `tenant`,`partition`)
|===

Alle Ressourcen tragen zusaetzlich: `app.kubernetes.io/instance`,
`app.kubernetes.io/part-of`, `tenant`, `partition`, `environment`,
`app.kubernetes.io/managed-by`.

== NetworkPolicy-Selektoren (Empfehlung)

* Primaer auf den Operator-Anker selektieren:
+
[source,yaml]
----
podSelector:
  matchLabels:
    cnpg.io/cluster: tifa01-cnpg-cluster
----
* Tenant-weite Policies ueber unsere eigenen (vererbten) Labels:
+
[source,yaml]
----
podSelector:
  matchLabels:
    tenant: tifa
    app.kubernetes.io/name: postgresql
----
* Pooler-Pods explizit beruecksichtigen (`cnpg.io/poolerName` bzw.
  `app.kubernetes.io/component: pooler`), ebenso Import-/Backup-Jobs.

== Mapping Spilo -> CNPG (fuer Migration der Policies)

[cols="2,2"]
|===
| Spilo/Zalando | CNPG-Aequivalent

| `application: spilo`        | `app.kubernetes.io/name: postgresql` (eigenes Label)
| `cluster-name: <name>`      | `cnpg.io/cluster: <name>`
| `spilo-role: master`        | `cnpg.io/instanceRole: primary`
| `spilo-role: replica`       | `cnpg.io/instanceRole: replica`
| `team: <team>`              | `tenant: <tenant>` / `partition: <partition>`
|===
