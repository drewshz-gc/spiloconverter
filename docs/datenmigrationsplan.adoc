== Architekturentscheidungen

=== Cluster-Modell: Monolith

**Entscheidung:** 1 CNPG-Cluster = 1 bisheriger Postmaster, alle `preparedDatabases` bleiben zusammen.

* `Database`-CRDs für jede zusätzliche DB (neben `bootstrap.initdb`)
* Kein Aufspaltung in 1.314 CNPG-Cluster
* Keycloak-DB bleibt im selben Cluster (nicht ausgelagert)

=== HA-Modell: 2 gleichwertige Instanzen

* `instances: 2`, `primaryUpdateMethod: switchover`
* **Kein** backup-only Replika (Condor-Lektion)
* Physisches Backup: `backup.target: prefer-standby`

=== Logical Backup (vertraglich)

* **Frequenz:** 1× wöchentlich (statt täglich/häufiger)
* **Mechanismus:** Kubernetes `CronJob`, Verbindung über `<cluster>-ro`
* **Secret:** außerhalb Git (ExternalSecrets / manuell)
* Spilo `enableLogicalBackup` entfällt am CNPG-Cluster

== Migrationsstrategien (pro Cluster)

[cols="1,2,2,2"]
|===
| Strategie | Wann | Downtime | Voraussetzung

| **WAL_RECOVERY** | Standard für Prod mit WAL-G | Minuten (Cutover) | S3-Archiv, Barman-kompatibel, `--s3-bucket`
| **INITDB + pg_dump/restore** | Klein, ohne WAL-G, Test | Proportional zur Größe | Wartungsfenster
| **Logical Replication** | Sehr große DBs, Zero-Downtime | Minimal | CNPG Subscription, PG-Upgrade ggf. nötig
|===

Empfehlung: **`WAL_RECOVERY`** für alle Cluster mit `USE_WALG_BACKUP=true`.

=== Ablauf WAL_RECOVERY (Standard)

. **Vorbereitung**
.. CNPG-Manifest generieren: `spilo2cnpg analyze` + `generate --strategy WAL_RECOVERY`
.. Barman `ObjectStore`/Credentials aus bestehendem WAL-G-Secret ableiten
.. ArgoCD-Manifeste auf Feature-Branch in `pcm-prod-apps`
. **Parallelbetrieb**
.. CNPG-Cluster bootstrapped aus S3-WAL-Archiv (noch kein Cutover)
.. Replikation einholen bis Lag ≈ 0
. **Cutover** (Wartungsfenster)
.. Apps stoppen / Read-only
.. Final WAL sync / letzter Checkpoint
.. Connection Strings auf CNPG `-rw` Service umstellen
.. Apps starten
. **Validierung**
.. Smoke-Tests pro DB
.. Keycloak-Realms prüfen
. **Abschluss**
.. Spilo-Cluster suspendieren / löschen (erst nach Retention)
.. Logical-Backup-CronJob aktivieren

== ArgoCD-Integration

* Manifeste in Tenant-`kustomization.yaml` (Feature-Branch)
* Sync-Waves: Cluster(0) → Database(1) → ScheduledBackup(2) → CronJob(3)
* Secrets **nicht** in Git
* `ignoreDifferences` auf `.status` für CNPG-CRDs
* Kein Deploy auf `main` ohne Review

== Rollback

| Phase | Maßnahme
| Cutover-Vorbereitung | Spilo läuft weiter, CNPG parallel — Rollback = DNS/Secret zurück
| Nach Cutover (<24h) | Apps auf Spilo `-rw` zurück, CNPG suspendieren
| Nach Cutover (>24h) | WAL_RECOVERY zurück auf Spilo oder Restore aus Logical/Physical Backup
|===

Spilo-Cluster **nicht löschen** bis CNPG 30 Tage produktiv + Backup verifiziert.

== Checkliste pro Cluster-Migration

=== Vor Migration

* [ ] `spilo2cnpg analyze` — keine BLOCKER

* [ ] Feature-Branch PR reviewed
* [ ] Wartungsfenster kommuniziert

=== Cutover

* [ ] Apps read-only / gestoppt

* [ ] Connection Strings / Tenant-CRD auf CNPG

=== Nach Migration

* [ ] Physisches Backup erfolgreich (ScheduledBackup)
* [ ] Manueller Logical-Backup-Job erfolgreich
* [ ] Monitoring/Alerts auf CNPG-Pods

== Werkzeuge

[source,bash]
----
# Analyse einzelner Cluster
spilo2cnpg analyze path/to/postgres-cluster.yaml

# CNPG-Manifest erzeugen
spilo2cnpg generate path/to/postgres-cluster.yaml \
  --strategy WAL_RECOVERY \
  --s3-bucket <bucket> \
  --argocd --sync-wave 0 \
  -o output/cnpg/cluster.yaml

# Bulk-Analyse (pcm-prod-apps checkout)
spilo2cnpg convert-all /path/to/pcm-prod-apps --analyze-only
----
